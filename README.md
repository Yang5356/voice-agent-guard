Voice Agent Guard

一个面向园区 / 写字楼门卫场景的 Voice Agent 产品原型。

本项目围绕“未登记访客拨打电话进行来访登记”这一真实业务场景展开，构建了一条从电话接入、自然对话采集、结构化抽取、记录存储，到门卫侧通知与处理的完整闭环。
在当前版本中，系统已经能够完成主链路跑通，并提供 Guard Dashboard 作为门卫侧工作台，支持查看记录、人工纠错、状态更新和基础查询。

一、项目目标与问题定义

这个项目的重点不是车牌识别硬件，也不是门禁控制本身，而是从“电话接通后的访客登记流程”切入，解决传统门卫人工登记效率低、信息易漏、处理链路分散的问题。

我对问题的定义是：

* 访客车辆未预登记时，司机通过电话与 Agent 交互
* Agent 以自然、简洁的方式采集关键信息
* 系统将结果结构化后推送到门卫侧
* 如果自动流程信息完整，则记录自动完成
* 如果信息不完整或存在错误，则门卫可在工作台中继续处理

当前聚焦的核心字段包括：

* 车牌号
* 来找谁
* 来做什么
* 手机号
* 入场时间（系统自动生成）

因此，这个项目真正要做的是一个电话驱动的 AI 门卫登记工作流系统，而不是单点功能 demo。

二、整体设计原则

本项目采用的不是“只够演示”的临时方案，而是：

先把核心闭环做对，再沿着可扩展路径逐步增强。

我在实现中坚持了几个原则：

1. 业务闭环优先

优先让“电话 → 对话 → 抽取 → 落库 → 通知 → 门卫处理”整条链路跑通，而不是先做局部炫技功能。

2. 半自建混合路线

不选择全部从零自研底层语音系统，也不把核心逻辑全部交给黑盒平台。
底层成熟能力可以复用，但关键业务规则必须自己掌控。

3. 自动化优先，人工兜底

系统目标不是让门卫重新做一遍机器做过的事，而是让 AI 尽可能自动完成，只有在不完整或识别错误时才由门卫接手。

4. 当前可落地，同时保留扩展性

每个技术选型都同时考虑了两个维度：

* 当前阶段是否能快速、稳定落地
* 后续是否具备继续增强或替换升级的空间

因此，我最终选择的路线不是“为了演示随便拼凑”，而是阶段性最优且具备延展性的实现路径。

三、技术选型与原因

1. 后端：FastAPI

我选择 FastAPI 作为后端主框架。

原因是这个项目本质上是一个 API 驱动的 AI workflow 系统，需要处理：

* webhook 回调
* JSON 输入输出
* 多模块协作
* 结构化字段处理
* Dashboard 路由
* 数据库存储与状态流转

相比 Flask，FastAPI 更适合当前这种：

* 接口多
* 结构化强
* 服务模块清晰
* 需要快速组织中控逻辑

的场景。

更重要的是，FastAPI 在本项目中承担的不是“文本处理工具”这么简单，而是整个系统的中控层与接口层。
它负责：

* 承接阿里云回调
* 提供前端页面和接口
* 组织 extraction / storage / notification 等服务模块
* 执行整通对话的最终结算逻辑
* 支撑门卫侧编辑与查询能力

所以，FastAPI 是整个产品原型的后端骨架，而不只是一个小工具。

2. 数据库：SQLite

我选择 SQLite 作为第一版数据库。

原因不是为了“图省事”，而是因为在当前阶段，它在以下方面达到了比较好的平衡：

* 本地开发与演示稳定
* 无需额外数据库服务
* 能提供完整的结构化存储能力
* 足以支撑当前阶段的记录管理和基础查询
* 后续也可以平滑升级到更完整的数据库方案

当前 SQLite 承载的内容包括：

* 来访记录
* transcript
* missing_fields
* 状态字段
* 后续门卫查询所依赖的历史数据

也就是说，SQLite 在当前不是“凑合”，而是一个对第一版产品原型非常合适的实现选择。
如果后续进入多用户、云部署或更复杂查询阶段，再升级 PostgreSQL 会更自然。

3. 字段抽取：OpenRouter + 大模型抽取

项目第一版先用规则抽取跑通最小闭环，后续引入 OpenRouter 模型抽取，用于提升真实自然语言场景下的结构化提取能力。

原因是电话场景中的用户表达经常会出现：

* 停顿
* 口误
* 前后更正
* 多轮分散表达
* 中文口语数字手机号
* 模糊表达后再补充更精确信息

这些场景下，纯规则方式很难稳定处理。
因此，大模型在这个项目里承担的是一个明确的角色：

不是为了“显得高级”而加 LLM，而是为了提高多轮自然语言信息抽取的稳定性。

为什么要做模型评估

OpenRouter 内部有很多模型，所以我没有随便选一个结束，而是围绕本项目场景做了实际评估。
我更关注的不是泛泛聊天能力，而是：

* 中文理解能力
* 结构化输出稳定性
* 对多轮合并文本的抽取表现
* 免费可用性与限流情况
* 与当前字段 schema 的匹配程度

因此，当前模型选择的逻辑不是“理论最强”，而是：

在当前业务任务中，找到效果、稳定性和可用性的实际平衡点。

4. 电话 / 语音链路：阿里云语音回调

我选择了 阿里云语音回调路径 来承接真实电话场景，而不是只停留在网页文本或浏览器语音模拟。

原因是：

* 题目明确要求真实拨号链路
* 当前方案能支撑真实电话接入验证
* 能让系统真正从“文字版原型”升级为“电话驱动工作流”
* 同时保留了后续继续向更完整呼入能力和更正式电话链路升级的空间

所以，阿里云在当前版本里并不是一个“一次性演示工具”，而是：

一条既能满足当前落地验证、又保留未来扩展能力的电话接入路径。

5. 微信通知：Server酱

我当前使用 Server酱 完成微信侧通知触达。

选择它的原因是：

* 微信触达直接
* 接入成本低
* 当前阶段能快速形成通知闭环
* 后续仍可通过会员 / 更完整方案增强能力
* 不会把大量时间消耗在重型企业通知接入上

这不是为了“临时演示凑一个微信消息”，而是因为它在当前阶段同时满足了两点：

* 可以快速形成真实通知能力
* 未来仍有进一步扩展空间

因此，当前通知链路不是死路。
如果后续继续演进，可以升级到更正式的企业微信、公众号或其他企业级通知通道。

四、系统架构

当前系统整体可以概括为：

访客电话
↓
阿里云语音回调
↓
FastAPI 后端
↓
回调聚合 / 字段抽取 / 状态判断 / 落库 / 通知
↓
SQLite + 微信提醒 + Guard Dashboard

后端内部主要模块包括：

* main.py：FastAPI 主入口、语音回调、页面路由、整通结算逻辑
* extraction_service.py：字段提取与结果规范化
* storage_service.py：数据库读写、编辑逻辑、门卫查询逻辑
* notification_service.py：门卫通知消息构造与微信推送
* models.py：访客记录表结构
* dashboard.html：门卫侧工作台页面

五、关键产品与工程决策

1. 从电话接通后的流程开始，而不是从车牌识别开始

我有意将项目范围聚焦在 Voice Agent 来电登记闭环，而不是门禁硬件或车牌识别。

原因是：

* 题目真正考察的是语音 AI 产品闭环
* 电话接通后的流程更能体现对话设计、抽取、状态管理和工程判断
* 从这里切入更容易形成完整且可展示的业务闭环

2. 将“25 秒”理解为主路径性能目标，而不是机械式硬中断

我没有采用“到 25 秒立刻挂断”的僵硬逻辑。
因为真实业务场景里，用户输入速度和配合程度并不完全可控。

更合理的产品化理解是：

* 主路径要尽量短
* 首轮尽量高密度采集信息
* 信息不完整时要有人工兜底，而不是假装自动化永远完美

在当前实现中，我加入了 60 秒保守兜底：

* 如果整通电话长时间未自然结束
* 系统会用当前已知信息落库
* 强制标记为 incomplete
* 推送到门卫侧继续人工补录

这个设计体现的是：

自动化优先，异常路径人工兜底

3. 采用整通对话聚合，而不是每句都立刻落库

我没有在每次回调句子到达时就直接写库。
而是按 instance_id 聚合同一通电话中的用户文本，直到：

* 检测到结束话术
* 或触发超时兜底

才统一做最终抽取和落库。

这样做的好处是：

* 提高抽取稳定性
* 支持多轮补充
* 支持前后修正
* 避免一句一句写库带来的脏记录

4. 过滤短确认语句，减少抽取噪音

像“对”“好的”“嗯”这类短确认，如果直接并入结构化抽取，容易干扰最终结果。
因此当前实现中会过滤这类短 ack，不把它们当成新的字段值参与合并。

5. 不做“只看通知”的门卫页，而是做可处理的 Guard Dashboard

如果系统只会“吐出一条消息”，它更像技术展示，而不像真实产品。

因此我增加了一个轻量版 Guard Dashboard，作为门卫侧工作台。
它目前支持：

* 查看最新来访
* 查看最近记录
* 展示状态
* 手动编辑完整字段
* 手动修改状态
* 基础自然语言查询

这个设计体现的思路是：

* AI 完成时，不额外增加人工动作
* AI 不完整或有误时，门卫可以顺畅接管

七、当前限制与未完成部分

1. 真正的回访识别尚未完成

题目里的加分项提到“回访识别”，理想情况是系统在电话刚接入时就能判断是否为老访客。
当前阿里云这条链路中，主叫号码并没有以足够直接、稳定的方式进入当前逻辑，因此我没有伪造这项能力。

也就是说，我更愿意明确写清当前限制，而不是在 README 中虚写一个并未真正实现的功能。

2. 门卫查询不支持“司机本人来了几次”

因为当前系统采集的是“来找谁”，而不是“访客本人姓名”。
因此“张师傅这个月来了几次”如果指的是司机本人，目前并不是准确支持的查询类型。

当前更合理的查询方向是：

* 被访对象维度
* 时间段维度
* 来访车辆数量维度

3. Dashboard 暂无登录 / 密码保护

当前 Guard Dashboard 作为本地原型工作台，尚未加入登录鉴权。
如果进入更正式部署阶段，后续应增加：

* 登录
* 密码保护
* 访问控制

4.关于多路并发

多路并发在技术上并非完全不可行，当前系统按 instance_id 对通话会话进行隔离与聚合，架构上具备进一步扩展的基础。
但在当前业务场景下，我没有把它作为第一版优先完成项，原因是它不仅是技术问题，还涉及现实流程设计问题，例如：

* 当前通话入口如何被后方车辆稳定触达，例如二维码 / 电话展示方式是否适合排队场景；
* 如果多辆车同时登记，门卫侧如何确保“哪辆车登记完成，就准确放行哪辆车”，避免放错对象；
* 并发场景下，通知、记录和线下放行动作之间需要更明确的对应机制。

因此，我当前优先完成的是单通主链路稳定闭环，而不是在缺少完整线下配套机制的情况下，先做一个看起来能并发但业务上并不闭合的版本。

八、我如何使用 AI

本项目开发过程中，我确实大量使用了 AI 辅助编码与辅助思考，但核心原则始终是：

用 AI 提高效率，不盲信 AI。

AI 在这个项目中帮助我做了很多事，包括：

* 辅助拆解系统模块
* 辅助生成代码骨架
* 辅助快速迭代后端和前端页面
* 辅助排查环境、依赖和接口问题
* 辅助整理设计记录与 README

但与此同时，我没有机械照搬 AI 的所有建议。
我自己做过多次关键判断，包括：

* 不从车牌识别开始做
* 不全靠 SaaS 黑盒
* 不把 25 秒理解为直接强制挂断
* 不伪装成已经做了真正 caller-ID 回访识别
* 不继续无止境扩展功能，而是在合适时点冻结进入提交准备
* 不让门卫对所有记录都重复确认，而是只在 incomplete 或人工纠错时介入

我理解这道题真正想考察的，并不只是“会不会写代码”，而是：

* 会不会做独立判断
* 会不会做 trade-off
* 会不会利用 AI，但不被 AI 牵着走

九、本地运行方式

安装依赖：

pip install -r requirements.txt

启动应用：

uvicorn app.main:app --reload --port 8000

打开门卫工作台：

http://127.0.0.1:8000/dashboard

十、环境变量

* LLM_API_URL
* LLM_API_KEY
* LLM_MODEL
* NOTIFY_MODE
* SERVERCHAN_SENDKEY

* LLM 抽取可通过 OpenRouter 路径完成
